# Hybrid Search 規則調參表

這份表把目前 `core/filters.py` 的實際規則整理成「核心詞 / 輔助詞 / 排除」三層，方便後續維護與調參。

## 1. 教育機構判定

### 會視為教育單位

| 類別 | 字詞 |
|---|---|
| 大專校院 | `大學`、`學院`、`科技大學`、`技術學院`、`專科學校`、`專科` |
| 軍校 | `軍官學校`、`國防大學`、`國防醫學院` |
| 中小學 | `學校`、`國小`、`國中`、`中學`、`高中`、`高職`、`附中`、`附小`、`附屬中學`、`附屬小學`、`附設實驗國民小學` |
| 教育行政 | `教育局`、`教育處` |

### 直接排除

| 類別 | 字詞 | 說明 |
|---|---|---|
| 強排除 | `附設醫院`、`附屬醫院`、`分院`、`診所` | 一律排除 |
| 弱排除 | `醫院` | 若機關名稱含 `醫學院` 可放行 |

### 白名單

| 字詞 | 用途 |
|---|---|
| `醫學院` | 覆蓋 `醫院` 的弱排除，保留教學單位 |

## 2. 核心詞規則

只有核心詞可以直接構成 IT 主題命中；輔助詞不能單獨算分。

### 核心詞群

| 類別 | 代表字詞 |
|---|---|
| AI / 資料分析 | `AI`、`人工智慧`、`AI 平台`、`AI運算`、`AI訓練`、`GPU`、`GPU算力`、`機器學習`、`深度學習`、`大數據平台`、`BI 系統`、`資料分析平台`、`生成式 AI` |
| 資安 | `資安`、`資通安全`、`防火牆`、`WAF`、`IDS`、`IPS`、`零信任`、`滲透測試`、`弱點掃描`、`SOC`、`SIEM`、`MFA`、`身分驗證`、`加密`、`EDR`、`XDR` |
| 網路 | `網路`、`網路設備`、`路由器`、`交換器`、`無線網路`、`WiFi`、`AP`、`VPN`、`網路管理`、`光纖`、`核心交換器`、`邊界交換器` |
| 系統 / 軟體 | `資訊系統`、`管理系統`、`校務系統`、`教務系統`、`學務系統`、`公文系統`、`人事系統`、`差勤系統`、`報修系統`、`單一登入`、`入口網站`、`ERP`、`CRM`、`APP`、`行動應用`、`軟體授權`、`Office 授權`、`Adobe 授權` |
| 雲端 / 機房 / 儲存 | `雲端`、`SaaS`、`IaaS`、`虛擬化`、`VM`、`VMware`、`Hyper-V`、`NAS`、`SAN`、`備份系統`、`異地備援`、`資料庫`、`SQL`、`機房`、`UPS`、`機櫃`、`伺服器`、`儲存設備` |

### 主判斷門檻

- `HIGH_CONFIDENCE_THRESHOLD = 2`
- 任一核心詞群命中即可直接形成高信心
- 英文縮寫採詞邊界比對，不再使用單純 `keyword in text`

## 3. 輔助詞規則

輔助詞不能單獨算分，只能搭配核心詞或明確 IT 場景形成 context match。

| 類別 | 字詞 | 建議用途 |
|---|---|---|
| 輔助詞 | `建置`、`導入`、`整合`、`開發`、`管理`、`平台`、`系統`、`文件`、`檔案`、`協作`、`智慧`、`分析`、`資料`、`決策` | 不可單獨作為 IT 判斷 |
| 搭配例子 | `建置校務系統`、`智慧校園管理平台`、`電子公文管理系統` | 有核心詞或明確 IT 場景時才形成命中 |

### 邊界門檻

- `LOW_CONFIDENCE_THRESHOLD = 0`
- 只有 context match 但未命中核心詞時，可保留為 `boundary`
- 這些案件才交給 embedding recall 做語意補強

## 5. 輸出標籤聚合

一案可有多個主題標籤，讓摘要聚合能反映複合型案件。例如 `機房電力及系統整合建置案` 會同時標示 `機房`、`電力`、`整合`，而不是只落在單一粗分類。

### 低分排除

| 字詞 | 用途 |
|---|---|
| `維修`、`耗材`、`消耗品` | 如果沒有其他資訊主題詞，會扣分並更容易落入排除 |

## 6. Embedding 補強規則

embedding 只處理邊界候選，不做第一層主篩選。

### 建議配置（2026 年更新）

#### 推薦模型：BAAI/bge-m3

```python
EMBEDDING_MODEL = "BAAI/bge-m3"
EMBEDDING_TOP_K = 30                   # 建議範圍 20-40
EMBEDDING_SIMILARITY_THRESHOLD = 0.68  # 建議範圍 0.65-0.72，從 0.68 開始測試
```

#### 為什麼選擇 BGE-M3？

| 優勢 | 說明 |
|------|------|
| **Hybrid Search 原生支援** | 單一模型同時輸出 dense（語意）+ sparse（類 BM25）+ ColBERT multi-vector，完美符合混合搜索架構 |
| **多語言優秀** | 支援 100+ 語言，在繁體中文 MIRACL zh 與長文件檢索（MLDR）上大幅超越 MiniLM |
| **長文本支援** | 上下文長度 8192 tokens，適合完整標案規格書或多頁 PDF |
| **推理效率高** | 568M 參數，可量化到 INT8/FP16，單張消費級 GPU 或 CPU 可跑 |
| **開源免費** | MIT 授權，完全免費自架，無 Azure/OpenAI token 費用 |

#### 使用方式

在 LlamaIndex / LangChain 中：
```python
from llama_index.embeddings import HuggingFaceEmbedding

embed_model = HuggingFaceEmbedding(model_name="BAAI/bge-m3")
```

模型會自動處理 dense + sparse 輸出，可直接餵給 Qdrant/Milvus 的 hybrid 模式。

#### 進階優化

- **搭配 CKIP 分詞**：sparse 部分效果更好（台灣專有名詞辨識準確）
- **參數微調**：可根據實際誤報/漏報情況調整 `SIMILARITY_THRESHOLD`
  - 若誤報多：提高至 0.70-0.72
  - 若漏報多：降低至 0.65-0.66

### 適合交給 embedding 的灰區例子

- `智慧教室`
- `教學平台`
- `文件流轉`
- `校務系統`
- `校園協作平台`
- `電子公文流程`

## 7. 維護建議

### 先加分、再觀察

- 新字詞先放進邊界層
- 觀察 1-2 次實際通知結果後，再決定是否升級成高信心

### 若誤報增加

- 先檢查是否有輔助詞 context 規則過寬
- 再檢查是否有新的排除詞需要補進 `EXCLUDE_KEYWORDS`

### 若漏報增加

- 先看是不是邊界詞不夠多
- 再看 embedding 的 `SIMILARITY_THRESHOLD` 是否需要微調

### 不建議直接升級成核心詞的字

- `智慧`
- `分析`
- `資料`
- `決策`
- `建置`
- `導入`
- `整合`
- `開發`
- `文件`
- `檔案`
- `平台`
- `系統`

這些詞很常出現在各種標案中，保留在輔助層比較安全。
